# 当前 App 耗电分析

## 结论范围

这是基于当前源码的静态分析，不等同于某一台设备上的实测电量。真实耗电还会受 SoC、网络制式、代理流量、屏幕亮度和系统后台策略影响。

## 场景矩阵

| 场景 | 当前主要工作 | 耗电风险 | 结论 |
| --- | --- | --- | --- |
| App 空闲、代理未启动 | UI 状态流、事件接收器、按需预览刷新；统计页时钟仅在有订阅者时每分钟更新 | 低 | 没有持续网络轮询 |
| App 前台、Tun/HTTP 运行 | 客户端流量查询、运行状态刷新；服务通知每 2 秒刷新（仅开启流量通知时） | 中 | 高频工作集中在用户正在查看 App 时 |
| App 退到后台、Tun/HTTP 运行 | native 核心持续转发；服务通知在亮屏且开启流量通知时每 2 秒刷新 | 中到高 | 转发耗电取决于实际流量；通知刷新会额外产生 IPC、查询和通知构建 |
| App 退到后台、屏幕熄灭、Tun/HTTP 运行 | native 核心持续转发；通知测速暂停；客户端状态轮询降为 15 秒 | 中 | 已避免无用户可见收益的高频 UI 轮询，核心网络耗电无法在不停止代理的情况下消除 |
| RootTun 运行 | RootTun 状态 IPC；亮屏约 4 秒、熄屏约 8 秒更新通知 | 中 | 状态通知轮询有固定 IPC/唤醒成本 |
| 代理组测速 | 每个目标发起健康检查；随后 3 次/4 次延迟刷新 | 高（短时） | 只应由用户主动触发；批量测速会产生多次网络握手和无线电活跃 |
| 配置下载/刷新 | 网络请求、配置解析、运行时重载 | 高（短时） | 受网络质量和配置大小影响，不是常驻耗电源 |
| 日志记录 | native 日志订阅、跨 JNI/进程传递、落盘 | 中 | 高日志量时会产生 CPU、序列化和 I/O；建议仅在诊断期间开启 |

## 真机验证方法

建议在同一设备、同一网络和同一代理配置下，各记录 30 分钟：无代理空闲、Tun 无流量、Tun 持续下载、亮屏首页、流量通知开关两种状态，以及 RootTun 与本地 Tun 两种模式。

可用 `adb shell dumpsys batterystats --reset`、`adb shell dumpsys batterystats <package>` 和 Battery Historian 对比；同时记录 `adb shell dumpsys cpuinfo`、`adb shell dumpsys netstats`，避免把网络本身的耗电误判为轮询开销。
